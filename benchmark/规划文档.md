# CAVE-VR Human Action Recognition Benchmark 规划文档

> 基于 Seminar_pres.pdf（2026-05）与 2026-07 文献调研整理。作者：Angus Zou / 整理：Claude。
> 数据现状：2 轮采集完成；第二轮为 6 面 CAVE、12 动捕 + 7 RGB 相机、15 演员、6 场景、7 通用 + 11 交互动作。

---

## 1. 相关工作与差异化定位

### 1.1 与现有数据集对比

| 数据集 | 规模 | 视角 | 模态 | 与你们的区别 |
|---|---|---|---|---|
| NTU RGB+D 120 (TPAMI 2020) | 106 人 / 120 类 / 114k clips | 正面/侧面，固定实验室 | RGB+D+骨架+IR | 无 VR 交互、无顶装视角、无实时约束 |
| ANUBIS (arXiv 2205.02071, PR 2026) | 80 人 / 102 类 / 66k clips | 40 前视 + 40 后视 | Azure Kinect 骨架(32关节) | 有后视但无顶装视角，无 VR 环境 |
| LocoVR (2024) | 多用户 VR 室内移动 | HMD 追踪 | 轨迹/姿态 | 只有移动(locomotion)，非动作分类 |
| HUMOTO (2025) | 人-物交互 4D mocap | — | mocap+物体 | 无 RGB 多视角、无 VR 场景 |
| RoCoG-v2 (2023) | 7 类手势，真实+合成 | 地面+空中 | RGB | 研究 domain gap 的思路可借鉴 |

### 1.2 你们的三个独特卖点（论文里要反复强调）

1. **顶装相机 + 交互遮挡**：现有 benchmark 几乎没有 top-mounted 视角；CAVE 中手/头/控制器遮挡是真实且系统性的，不是人为制造的。
2. **VR 交互动作类别**：controller 移动、捕鱼、抓箱、射击等 11 类交互动作在通用数据集中不存在。
3. **动捕级 3D 骨架真值 + 多视角 RGB 同步**：12 台 Primex 13 提供的骨架精度远高于 Kinect 类数据集（NTU/ANUBIS 都是深度相机估计骨架），这是质量优势。
4. （可选第四点）**实时/延迟评测维度**：现有分类 benchmark 只报准确率；你们可以是第一个为 VR 场景定义 latency-aware 协议的 HAR benchmark。

### 1.3 定位一句话

"首个在 CAVE 式 VR 环境中采集、带动捕级骨架真值和多视角 RGB 的 HAR benchmark，用于量化通用 HAR 模型到 VR 场景的 domain gap，并引入延迟感知评测。"

## 2. Benchmark 设计建议

### 2.1 数据划分（split）

15 人的规模决定了 split 要谨慎设计，避免评测方差过大：

- **Cross-Subject (X-Sub)**：主协议。建议 10 人训练 / 5 人测试（保持男女比例接近 2:1），并报告 3 组不同 subject 划分的均值±方差 —— 人少时单一划分结果不稳定，多划分平均是审稿人会认可的做法。
- **Cross-View (X-View)**：用 7 个 RGB 机位做训练/测试视角分离，如高位训练→低位测试、正面训练→对角后视测试。这是展示"顶装视角难度"的核心实验。
- **Cross-Scene (X-Scene)**：3 室内训练 / 3 室外测试（VR 投影内容变化 = 背景/光照 domain shift），这是 CAVE 独有的实验条件——真实世界数据集做不到"同一动作、完全可控地换背景"。
- **Cross-Domain（NTU → 你们）**：在 NTU RGB+D 120 上预训练的模型直接在你们测试集上 zero-shot / fine-tune 评测，量化 domain gap。这一实验直接支撑论文动机。

### 2.2 评测指标

- 分类：Top-1 / Top-5 accuracy + **mean per-class accuracy**（18 类且交互动作样本可能不均衡，必须报 per-class）。
- 混淆矩阵重点分析：通用动作 vs 交互动作、遮挡严重视角 vs 正常视角。
- **延迟维度（差异化亮点）**：借鉴 Online Action Detection 的做法，报告两个量：
  - *computation latency*：模型单帧推理耗时（固定硬件，如单张消费级 GPU）；
  - *observation latency*：模型需要观察多少帧才能正确分类（对 VR 实时交互至关重要）。
  - 可画 accuracy-vs-latency 曲线，给每个 baseline 一个"在 X ms 预算内的准确率"。

### 2.3 Baseline 计划与分工（2026-07 确定）

**学生 A — Skeleton 组（GCN 系列，输入格式统一 N×C×T×V×M）**

| 模型 | 模态 | 备注 |
|---|---|---|
| ST-GCN | Skeleton | 最经典，先跑通验证数据 |
| CTR-GCN | Skeleton | ST-GCN 升级版，代码结构几乎一样 |
| SkateFormer | Skeleton | 输入格式与上面一致 |
| SkelMamba | Skeleton | 同上 |

⚠️ 共同前置工作：这些 repo 默认 NTU-25 关节拓扑，**你们的 SMPL-22/24 关节集需要自定义邻接矩阵（graph 定义）**，一次定义、四个模型复用。归一化约定（pelvis 居中、朝向对齐）也要统一写进共享 dataloader。

**学生 B — RGB 组（视频模型，dataloader 复用）**

| 模型 | 模态 | 备注 |
|---|---|---|
| SlowFast | RGB | MMAction2 原生支持 |
| VideoMAE v2 | RGB | MMAction2 或官方 repo |
| VideoMamba | RGB | 输入 pipeline 同上 |
| Video Swin | RGB | MMAction2 |

⚠️ 建议：每个相机视角作为独立样本并保留 view 标签（X-View 协议需要）；先做一份统一的 video/rawframes 标注列表，四个模型共用。

**Angus — 核心 + 高难度模型**

| 模型 | 模态 | 备注 |
|---|---|---|
| ActionCLIP | RGB+Text | zero-shot + finetune，需为 18 类动作撰写 Action Description；zero-shot 结果本身是论文亮点 |
| PoseConv3D | Skeleton+RGB | 双流融合。它要的 2D pose heatmap 可以**直接用修正后的 3D 真值投影到各相机生成**——比姿态估计的 heatmap 干净，这是动捕真值的独家优势，论文里值得强调 |
| MotionBERT | Skeleton | Foundation model，要求 H36M-17 关节布局，需再做一次关节映射 + 其自有归一化；调参复杂 |

**三人共享基建（保证数字可比，先做）**：统一 split 文件、统一 label map、统一结果记录格式（CSV：model / modality / split / top1 / top5 / mean-class-acc / seed）、固定随机种子。没有这个，三个人的结果拼不进同一张表。

（可选补充：ProtoGCN 为当前 NTU 120 SOTA，若时间允许可加入 Skeleton 组；EPAM-Net 可作为额外融合基线。）

## 3. 数据处理方案

### 3.1 多视角同步（当前最大痛点）

无硬件 genlock 时的实用方案，按优先级：

1. **音频对齐**：所有 RGB 相机若带麦克风，录制开始/结束各拍一次响板（clap），用音频互相关自动对齐，精度可到 ~1 帧。这是文献中最常用的离线同步方法。注意：闪光灯不可靠——rolling shutter 会导致闪光只出现在部分扫描行上。
2. **动作事件对齐**：若无音频，让演员每段开头做一个尖锐动作（如双手快速击掌），用 mocap 骨架速度峰值和各视角 2D 关键点速度峰值做互相关。可写脚本批量处理，不必逐段手动。
3. **Mocap 为主时钟**：OptiTrack（Primex 13 配 Motive）自带精确时间戳，把所有 RGB 流对齐到 mocap 时间轴，RGB 相机之间不需两两对齐。
4. 未来第三轮采集建议：买支持外触发的工业相机或加一个 timecode 发生器，一次性解决。

### 3.2 标定偏移处理

- 对每个录制日维护独立标定集（你们已在做），并给每段数据记录"所属标定版本"的 metadata。
- 用 **重投影误差** 做自动质检：把 mocap 3D 关节投影到各 RGB 视角，与 2D 姿态估计（如 RTMPose）结果比对，误差超阈值的 session 标记需要重标定或丢弃。这比人工目检快且可量化，论文里还能作为数据质量指标报告。

### 3.3 骨架格式统一

- 37 marker → 关节：Motive 输出骨骼后，映射到一个标准关节集。建议 **主格式取 NTU-25 兼容布局**（方便直接跑 ProtoGCN/CTR-GCN 等现成代码），保留原始高精度数据作为附加格式。
- 已有文献做法：NTU-25 与 SMPL 有 19 个关节直接对应，缺失关节用相邻关节中点插值（spine1 = SpineBase–SpineMid 中点等）。你们从 mocap 出发更简单——只需下采样关节，不需插值。
- 坐标系统一：所有骨架转到以地面为 XY、人体初始朝向对齐的 CAVE 世界坐标系；每段序列存储原始世界坐标 + 归一化坐标两份。

### 3.3b 骨架修正策略（2026-07 更新）

经实测，2D 姿态模型预测回填 SMPL-22 关节效果不佳（错误关节、严重抖动），**确定采用人工关键帧 + 平滑插值为主的质量优先路线**（CVSlice Skeleton Corrector）。自动化只用于"帮人找问题"，不直接改数据：

1. **坏帧自动导航**：用 `detect_bad_frames` / 骨长突变 / 速度尖峰打分，热键跳到下一个可疑帧——省掉逐帧 A/D 排查的时间，修不修仍由人决定；
2. **片段级 QC 排序**：批量给所有片段打质量分，人按"最差优先"顺序修，好的片段抽查即可，不必平均用力；
3. **工程加速**：2D 检测结果按片段缓存、onnxruntime-gpu 推理——只影响等待时间，不影响数据。

论文表述："all corrections are manually keyframed and verified; automated detection is used only to prioritize review"——这句比任何自动修正都让审稿人放心。

### 3.4 清洗 pipeline（建议顺序）

1. 按重投影误差 + marker 丢失率自动打分，剔除损坏段；
2. 时间同步（3.1）；
3. 动作分割与标签校验（两人独立标注抽样 10% 计算一致率，写进论文）;
4. 格式转换与坐标归一化（3.3）；
5. 生成 split 文件与统计表（每类样本数、时长分布）。

## 4. 论文规划

### 4.1 目标 venue（按时间线：2026.6–9 baseline，10 月起写作）

- **CVPR 2027**（deadline 通常 2026 年 11 月中）：时间最匹配，主会收 dataset+benchmark 论文，但竞争激烈，需要完整 baseline + 分析。
- **NeurIPS 2027 Evaluations & Datasets track**（deadline 约 2027 年 5 月）：2026 届截稿是 5 月 6 日，已过；投 2027 届时间宽裕，且该 track 明确要求"数据集要服务于评测方法"——你们的 latency-aware 协议正好契合。
- **IEEE VR / ISMAR 2027**：VR 社区对 CAVE 工作接受度高，竞争小于 CVPR，可作为并行选项。
- 期刊备选：IJCV / TPAMI dataset paper（周期长）。
- ⚠️ 具体 deadline 请在投稿前 2-3 个月到官网核实，以上为基于历史规律的估计。

### 4.2 Dataset paper 结构草案

1. **Introduction**：VR 训练/交互需要 HAR → 现有数据集三个假设（固定视角、无遮挡、离线）在 CAVE 中全部失效 → 我们的贡献（数据集 + domain gap 量化 + latency 协议）。
2. **Related Work**：HAR 数据集（NTU/ANUBIS/Kinetics）、VR 用户行为数据集（LocoVR 等）、online action detection。
3. **The XXX Dataset**（起个名字，如 CAVE-HAR / HKU-VRAct）：采集环境、相机布置图、标定流程、同步方案、动作类别定义、统计表、与现有数据集对比表。
4. **Benchmark Protocol**：X-Sub / X-View / X-Scene / Cross-Domain 四个 split + 指标 + latency 协议。
5. **Experiments**：baseline 结果、domain gap 分析、遮挡视角消融、失败案例可视化。
6. **Limitations & Ethics**：15 人规模、单一 CAVE 场地、演员知情同意与数据发布许可（**现在就确认所有演员签过可公开发布的同意书，含 RGB 视频肖像权** —— 这是 dataset paper 被拒的常见硬伤）。

### 4.3 近期行动清单（2026年7月）

1. 确认演员数据发布授权是否覆盖公开 benchmark；
2. 跑通同步脚本（3.1 方案 2/3），估算全量处理工时；
3. 先在小子集（2-3 人）上把 ProtoGCN 和 SlowFast 跑通，验证格式转换正确，再全量处理；
4. 给数据集起名并注册项目主页/GitHub 仓库。
